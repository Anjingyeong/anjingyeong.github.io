# Saramin Human Tone Check

## 점검 기준

이 문서는 사람인 이력서에 넣기 전 마지막으로 확인할 표현을 정리한 파일입니다. 공개 포트폴리오와 맞지 않는 과장, 확인되지 않은 수치, 내부 분석용 표현, 너무 정리된 느낌의 문장을 줄이는 데 목적이 있습니다.

## 과장되거나 AI스럽게 보일 수 있는 표현

| 피해야 할 느낌 | 더 자연스러운 대체 문장 |
| --- | --- |
| 제품이 이미 완성됐다고 단정하는 느낌 | 실시간 영상 AI 관제 흐름을 설계하고, 서비스 적용을 고려한 이벤트 전달 구조를 정리했습니다. |
| 근거 없이 성능을 크게 부풀리는 느낌 | 기존 포트폴리오 자료 기준으로 성능 개선을 확인했고, 재현 가능한 근거를 함께 확인할 필요가 있습니다. |
| 기술을 지나치게 새롭고 특별하게 포장하는 느낌 | 프로젝트 문서와 실험 기록을 다시 찾을 수 있도록 RAG 기반 지식화 구조를 정리했습니다. |
| 문제를 전부 해결했다고 단정하는 느낌 | 주요 문제를 분석하고, 개선 방향과 후속 검증 항목을 정리했습니다. |
| 경험을 넓게만 말하고 실제 작업이 보이지 않는 느낌 | 영상 AI, 데이터 분석, 백엔드 연동 프로젝트를 통해 모델 결과를 서비스 흐름으로 연결하는 경험을 쌓았습니다. |
| 도구 활용을 과하게 포장하는 느낌 | AI 도구는 반복 작업과 초안 작성에 활용하고, 복잡한 오류는 로그와 공식 문서를 기준으로 직접 확인합니다. |

## 포트폴리오와 충돌하지 않게 조정한 부분

### 스마트 안전 관제 시스템

- 대표 프로젝트로 가장 앞에 배치했습니다.
- 실시간 영상 AI, 이벤트 파이프라인, overlay 동기화, FP/FN 분석 방향을 강조했습니다.
- 검증되지 않은 Recall, Precision, F1, FPS, 지연시간 수치는 넣지 않았습니다.
- "완성된 제품"처럼 보이는 표현 대신 "구조 설계", "실험 방향 정리", "서비스 적용을 고려한 흐름"으로 조정했습니다.

### LLM Wiki/RAG 프로젝트

- 보조 대표 프로젝트로 배치했습니다.
- 프로젝트 문서, 실험 기록, 의사결정 근거를 검색 가능한 지식 자산으로 정리한 경험을 강조했습니다.
- GraphRAG, re-ranking, 고급 검색 전략은 실제 적용이 확실하지 않은 부분이므로 "고도화 방향" 또는 "검토한 구조" 수준으로 표현했습니다.

### RF-DETR 대장 용종 탐지

- Computer Vision 직무 연관성이 높아 주요 프로젝트로 유지했습니다.
- mAP 개선과 22 FPS 표현은 기존 포트폴리오 자료 기준으로만 사용했습니다.
- 최종 제출 전 학습 로그, 발표 자료, GitHub README 중 하나에서 수치 근거를 다시 확인하는 것이 좋습니다.

### VAE 유방암 이상 탐지

- 의료 영상 이상 탐지와 데이터 분석 경험을 보여주는 보조 프로젝트로 정리했습니다.
- Dice 약 90%와 수상 이력은 기존 포트폴리오 자료 기준으로 사용했습니다.
- 공모전명, 수상명, 기간은 사람인 입력 전 원문 자료와 대조해야 합니다.

### FullCount / Lumina Capital / 마음이음

- AI Engineer 이력서에서는 FullCount만 서비스 연동 경험 보조 사례로 짧게 반영했습니다.
- Lumina Capital은 필요 시 데이터 기반 화면 구현 경험 정도로만 사용할 수 있습니다.
- 마음이음은 메인 프로젝트로 강조하지 않고, 배포형 서비스 경험이 필요할 때만 짧게 쓰는 방향이 적절합니다.

## 민감정보 점검

현재 생성 문서에는 비밀번호, 인증 토큰, 서버 접속 정보, 데이터베이스 비밀번호를 넣지 않았습니다. 다만 GitHub 링크를 추가할 때 private 저장소 URL, 초대 링크, 내부 서버 주소가 들어가지 않도록 확인해야 합니다.

## 최종 제출 전 추천 수정

1. 사람인 프로젝트 입력란이 짧으면 스마트 안전 관제, LLM Wiki/RAG, RF-DETR 3개만 우선 입력합니다.
2. Computer Vision 직무에는 VAE를 추가하고, 서비스형 AI 직무에는 FullCount를 추가합니다.
3. 자기소개 문항은 800자 버전을 기본으로 쓰고, 글자 수 제한이 작으면 500자 버전을 사용합니다.
4. "직접 구현", "설계", "검토", "정리"를 실제 작업 범위에 맞춰 마지막으로 조정합니다.
5. 수치가 들어간 문장은 제출 전 근거 파일을 열어 한 번 더 확인합니다.

## 문장 톤 판정

- 문장이 너무 논문식으로 흐르지 않도록 "모델", "데이터", "서비스 흐름" 중심으로 정리했습니다.
- 신입 지원자 톤에 맞게 "검증했습니다", "정리했습니다", "개선 방향을 도출했습니다"를 사용했습니다.
- 결과를 과장하지 않기 위해 스마트 안전 관제 프로젝트에는 확인되지 않은 정량 성과를 넣지 않았습니다.
- 공개 포트폴리오와 이력서가 서로 다른 이야기를 하지 않도록 대표 프로젝트 순서를 맞췄습니다.
