# Saramin Self Introduction

## 500자 버전

[모델 결과를 서비스 흐름까지 연결하는 AI Engineer]

AI 모델은 성능 지표를 높이는 것만큼 **실제 환경에서 안정적으로 쓰이는 구조**가 중요하다고 생각합니다. 의공학 캡스톤에서는 **Kvasir 대장 용종 데이터**를 Train 70% / Validation 20% / Test 10%로 분할하고, RF-DETR 기반 탐지 모델을 의료 데이터에 맞게 fine-tuning했습니다. 이후 스마트 안전 관제 프로젝트에서는 **YOLO Pose + LSTM 기반 이상행동 탐지 흐름**을 정리하고, RTSP 입력부터 MQTT/WebSocket 이벤트 전달까지 서비스 관점의 파이프라인을 설계했습니다. FP/FN 사례를 보며 어떤 데이터를 다시 모아야 하는지도 함께 고민했습니다. LLM Wiki/RAG 프로젝트에서는 문서와 실험 기록을 다시 찾을 수 있는 지식 자산으로 구조화했습니다. 앞으로도 모델을 만드는 데서 끝내지 않고, **데이터와 운영 흐름을 함께 보는 AI Engineer**로 성장하고 싶습니다.

## 800자 버전

[실험 결과를 실제 서비스 흐름으로 옮기는 개발자]

저는 AI 모델을 만드는 일에서 가장 중요한 부분이 "좋은 결과가 실제 사용 흐름 안에서 어떻게 쓰이는가"라고 생각합니다. 의공학을 전공하며 의료 영상 데이터를 다뤘고, 캡스톤 프로젝트에서는 **RF-DETR 기반 대장 용종 탐지 모델**을 개발했습니다. Kvasir 대장 내 용종 이미지를 Train 70% / Validation 20% / Test 10%로 분리해 학습·검증·평가 흐름을 구성했고, 내시경 환경을 고려한 데이터 증강과 fine-tuning을 통해 최고 성능 mAP@50 86.2%(베이스라인 대비 약 7%p 개선) 및 22+ FPS의 실시간 추론 성능을 확보했습니다.

부트캠프 이후에는 AI 모델과 서비스가 연결되는 구조에 더 관심을 두었습니다. 스마트 안전 관제 프로젝트에서는 **YOLO Pose로 keypoint를 추출하고 LSTM으로 시계열 패턴을 분류하는 흐름**을 정리했습니다. 또한 RTSP 영상 입력, MQTT/WebSocket 이벤트 전달, 관제 화면 표시 구조를 나누어 보며 AI 결과가 사용자 화면까지 전달되는 과정을 설계했습니다. frameId, timestamp, cameraId, confidence 같은 이벤트 메타데이터를 함께 고려했고, FP/FN 분석과 hard negative 재학습 후보 정리처럼 모델 개선에 필요한 데이터 흐름도 함께 고민했습니다.

저는 아직 더 검증하고 쌓아야 할 부분이 많다고 생각합니다. 대신 프로젝트에서 한 일을 과장하기보다, 어떤 문제가 있었고 어떤 기준으로 개선하려 했는지를 분명히 설명할 수 있는 개발자가 되고 싶습니다. 팀 프로젝트에서 CORS, API 규격 불일치, Git 브랜치 충돌을 겪으며 **로그와 요청 흐름을 직접 확인하는 습관**도 생겼습니다. 모델, 데이터, 백엔드, 화면이 맞물려 돌아가는 흐름을 이해하는 AI Engineer로 성장하겠습니다.

## 1000자 버전

[AI 결과를 사람이 쓸 수 있는 서비스로 연결하고 싶습니다]

AI Engineer를 목표로 하게 된 이유는 모델의 결과가 실제 사용자에게 전달되는 과정에 관심이 많았기 때문입니다. 처음에는 모델 성능을 높이는 데 집중했지만, 프로젝트를 진행할수록 좋은 지표만으로는 충분하지 않다는 것을 느꼈습니다. 입력 데이터가 어떤 환경에서 들어오는지, 결과가 어떤 형태로 저장되고 전달되는지, 사용자는 그 결과를 어떻게 확인하는지가 함께 맞아야 프로젝트가 완성된다고 생각합니다.

의공학 캡스톤 프로젝트에서는 **RF-DETR 기반 대장 용종 탐지 모델**을 개발했습니다. Kvasir 대장 내 용종 이미지를 Train 70% / Validation 20% / Test 10%로 분할해 학습, 검증, 평가 흐름을 나누었고, 데이터 전처리와 모델 학습, fine-tuning을 담당했습니다. 의료 영상 데이터는 병변 크기와 형태가 다양해 단순한 이미지 증강만으로는 한계가 있다고 판단했습니다. 그래서 내시경 환경에 맞는 변형과 노이즈를 반영하는 방식으로 데이터를 보강했고, 후보 모델 및 FPS 비교를 통해 실시간 판독 보조 시스템에 필요한 처리 조건도 함께 확인했습니다.

최근에는 스마트 안전 관제 시스템을 대표 프로젝트로 정리하며 **실시간 영상 AI 파이프라인**을 설계했습니다. YOLO Pose로 사람의 keypoint를 추출하고 LSTM으로 시간 흐름에 따른 이상행동을 분류하는 구조를 잡았으며, RTSP 입력, MQTT/WebSocket 이벤트 전달, 관제 화면 표시까지 이어지는 흐름을 나누어 정리했습니다. 특히 frameId와 timestamp를 기준으로 영상 증거와 AI 이벤트가 어긋나지 않도록 overlay 동기화 구조를 검토했고, FP/FN 분석을 통해 hard negative 재학습 후보를 정리하는 방식도 함께 고민했습니다. 정량 지표인 Precision, Recall, F1-score, FPS는 실험 로그 확인 후 보강할 예정입니다. VAE 기반 유방 초음파 이상 탐지 프로젝트에서는 라벨이 부족한 의료 영상 문제를 **비지도 이상 탐지**로 접근했습니다. 제가 맡은 범위는 유방 초음파 데이터 전처리와 Reconstruction Error Map 생성, Dynamic Threshold 후처리 알고리즘 개발이었습니다. 고정 임계값만으로는 환자별 밝기와 노이즈 편차에 대응하기 어렵기 때문에, 영상별 픽셀 분포를 반영해 병변 후보 영역을 보정하는 방식으로 오탐을 줄이고자 했습니다.

LLM Wiki/RAG 프로젝트에서는 프로젝트 문서와 실험 기록을 다시 찾을 수 있는 지식 자산으로 만드는 방법을 다뤘습니다. 단순히 문서를 많이 쓰는 것이 아니라, 문제 정의, 의사결정, 실험 근거가 나중에 이력서나 면접 답변으로 이어질 수 있도록 **BM25, Vector Search, Metadata Filtering, RRF 기반 검색 흐름**으로 구조화했습니다.

부트캠프 팀 프로젝트에서는 프론트엔드와 백엔드를 분리해 연동하면서 CORS, API 규격 불일치, Git 브랜치 충돌 같은 문제를 겪었습니다. 처음에는 AI 도구에 에러 로그를 붙여넣고 답을 기대한 적도 있었지만, 결국 시스템 흐름을 이해하는 것은 제 몫이라는 것을 배웠습니다. 이후에는 공식 문서와 로그를 먼저 확인하고, 데이터가 어디에서 어떻게 바뀌는지 직접 추적하려고 합니다. 앞으로도 AI 도구는 속도를 높이는 보조 수단으로 활용하되, **문제의 원인을 끝까지 확인하는 태도**를 잃지 않겠습니다.

## 문항별 조합용 문단

### 지원동기

AI 모델을 만드는 것뿐 아니라, 그 결과가 실제 서비스 안에서 어떻게 전달되고 사용되는지에 관심이 있습니다. 스마트 안전 관제 프로젝트에서 **RTSP 입력, 모델 추론, MQTT/WebSocket 이벤트 전달, 관제 화면 표시** 흐름을 함께 정리하며 AI Engineer에게 필요한 역량이 모델 개발과 시스템 이해를 함께 요구한다는 점을 배웠습니다.

### 직무 역량

YOLO Pose, LSTM, RF-DETR, VAE 기반 프로젝트를 통해 영상 데이터 처리, 객체 탐지, 이상 탐지, 시계열 분류 흐름을 경험했습니다. 특히 RF-DETR 프로젝트에서는 Train 70% / Validation 20% / Test 10% 분할과 fine-tuning을 수행했고, VAE 프로젝트에서는 **데이터 전처리와 Dynamic Threshold 후처리**를 담당했습니다. 성능 수치를 단순히 높이는 것보다 FP/FN 사례를 확인하고, 데이터 보강과 재학습 후보를 정리하며 개선 방향을 찾는 방식에 익숙합니다.

### 협업 경험

부트캠프 팀 프로젝트에서는 프론트엔드와 백엔드 연동 과정에서 API 규격 불일치, CORS, Git 브랜치 충돌을 경험했습니다. 문제를 빠르게 넘기기보다 **요청과 응답 구조를 확인하고 팀원과 API 명세를 맞춰가는 방식**으로 해결했습니다. 정확한 팀 규모와 프로젝트 기간은 제출 전 최종 확인이 필요합니다. [수치 확인 필요]
