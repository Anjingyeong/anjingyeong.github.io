# Project STAR-RN Stories

## 1. 스마트 안전 관제 시스템

### 포트폴리오 상세 버전

**S:** CCTV는 많지만 관제자가 모든 화면을 24시간 동일한 집중도로 확인하기 어렵습니다. 단순 탐지 모델이 아니라 실시간 관제 보조 시스템으로 동작해야 했기 때문에, 영상 프레임과 AI 이벤트, 대시보드 알림이 일치해야 했습니다.

**T:** AI 파트에서 이상행동 탐지 모델과 실시간 추론 파이프라인을 설계/구현하고, 오탐/미탐을 분석해 실서비스 환경에서의 안정성을 검증하는 것이 목표였습니다.

**A:** YOLO Pose 기반 keypoint 추출, LSTM 기반 시계열 행동 분류, RTSP/WebRTC/MQTT 기반 실시간 파이프라인을 핵심 구조로 정리했습니다. 영상과 event metadata를 분리하고, frameId/timestamp/evidence chain, FP/FN 분석, hard negative 후보, threshold/tracker/frame drop/active_tracks 분석을 운영 안정성 기준으로 묶었습니다.

**R:** 현재 확인된 범위에서는 성능 수치를 임의로 작성하지 않습니다. 대신 RTSP -> AI 추론 -> MQTT 이벤트 -> WebSocket 알림으로 이어지는 구조와, overlay 동기화 및 evidence 추적 관점을 포트폴리오 문서로 정리했습니다.

**R:** 이 경험을 통해 AI 모델 성능만으로는 실제 서비스 품질을 보장할 수 없고, 영상 프레임과 이벤트 증거가 일치해야 신뢰 가능한 관제 시스템이 된다는 점을 배웠습니다.

**N:** 실제 승인된 real error 후보 기반 hard negative 재학습, frame_sync payload 고도화, observability 도입, 운영 환경별 threshold 정책 검증을 다음 단계로 둡니다.

### 면접 1분 버전

스마트 안전 관제 프로젝트에서는 CCTV를 사람이 계속 확인하기 어렵다는 문제를 실시간 AI 관제 보조 시스템 문제로 재정의했습니다. 단순히 낙상 탐지 모델을 만드는 데서 끝내지 않고, RTSP 영상 입력, AI 추론, MQTT 이벤트, WebSocket 대시보드 알림이 이어지는 구조를 정리했습니다. 특히 frameId와 timestamp 기반 evidence chain, FP/FN 분석, threshold와 tracker 상태처럼 운영 안정성에 영향을 주는 요소를 핵심 과제로 봤습니다. 이 경험을 통해 AI 모델 성능만큼 이벤트 증거와 화면 동기화가 중요하다는 점을 배웠고, 다음 단계로 hard negative 재학습과 observability 고도화를 계획하고 있습니다.

### 자기소개서 문단 버전

저는 스마트 안전 관제 시스템을 단순한 낙상 탐지 모델이 아니라, 관제자의 판단 공백을 줄이는 실시간 AI 보조 시스템으로 바라봤습니다. RTSP 영상 입력, AI 추론, MQTT 이벤트 발행, WebSocket 알림이 하나의 흐름으로 연결되어야 실제 서비스에서 의미가 있다고 판단했고, frameId/timestamp 기반 evidence chain과 FP/FN 분석, hard negative 후보 설계를 중요한 개선 축으로 정리했습니다. 이 경험을 통해 AI 프로젝트에서는 모델 정확도뿐 아니라 이벤트 증거, 화면 동기화, 운영 안정성까지 함께 설계해야 한다는 관점을 갖게 되었습니다.

## 2. LLM Wiki / Evidence Wiki / RAG

### 포트폴리오 상세 버전

**S:** 프로젝트가 복잡해지면서 실험 결과, 오류 해결, 의사결정 기록이 여러 도구와 폴더에 흩어졌습니다.

**T:** 개발 과정에서 생성된 지식을 검색 가능하고 재사용 가능한 형태로 정리하는 내부 지식 시스템을 구축해야 했습니다.

**A:** 문서 기반 Wiki를 구성하고, RAG 검색 구조와 Hybrid Search 방향을 정리했습니다. 실험 결과와 트러블슈팅 기록을 evidence로 축적하고, 포트폴리오/면접 답변으로 재사용 가능한 문서 구조를 만들었습니다.

**R:** 프로젝트 기록을 단순 메모가 아니라 지식 자산으로 전환했습니다. 오류 해결과 실험 근거를 빠르게 재참조할 수 있는 기반을 마련했고, 포트폴리오 정리에도 활용했습니다.

**R:** AI 프로젝트에서는 코드뿐 아니라 의사결정 근거와 실패 기록이 중요하다는 점을 배웠습니다. RAG는 단순 챗봇이 아니라 개발 지식 관리 도구가 될 수 있습니다.

**N:** metadata filtering, BM25, vector search, RRF, optional re-ranking을 고도화하고, 문서 관계가 커질 경우 GraphRAG를 확장 후보로 검토합니다.

### 면접 1분 버전

LLM Wiki 프로젝트는 개발 기록이 흩어지는 문제에서 출발했습니다. 실험 결과, 오류 해결, 의사결정 근거를 나중에 다시 찾기 어렵다고 느껴 문서 기반 Wiki와 RAG 검색 구조를 설계했습니다. 단순 챗봇이 아니라 포트폴리오와 면접 답변의 evidence repository로 사용할 수 있도록 source와 검증 기록을 함께 남기는 방식입니다. 이 경험을 통해 AI 프로젝트에서는 코드뿐 아니라 실패 기록과 의사결정 근거가 중요한 자산이 된다는 점을 배웠습니다.

### 자기소개서 문단 버전

AI 프로젝트를 진행하면서 저는 코드만큼이나 실험 실패, 오류 해결, 의사결정 근거가 중요하다는 점을 체감했습니다. 이를 해결하기 위해 LLM Wiki와 RAG 기반 프로젝트 지식 시스템을 정리했고, 흩어진 기록을 검색 가능하고 재사용 가능한 evidence로 전환했습니다. 이 경험은 포트폴리오를 과장된 결과물이 아니라 근거 기반의 이야기로 재구성하는 데 도움이 되었고, 개발 지식 관리 도구로서 RAG를 바라보는 관점을 갖게 했습니다.

## 3. RF-DETR 실시간 대장 내 용종 검출

### 포트폴리오 상세 버전

**S:** 대장 내시경 영상은 조명 변화, 반사광, 장벽 왜곡, 병변 형태 다양성으로 인해 일반 객체 탐지 모델만으로는 안정적인 검출이 어렵습니다.

**T:** 정확도와 실시간 추론 속도를 동시에 만족하는 의료 영상 객체 탐지 파이프라인을 구성해야 했습니다.

**A:** RF-DETR 기반 추론 구조를 사용하고, Grid Distortion과 Elastic Deform으로 대장 내벽 왜곡을 모사하는 데이터 증강을 적용했습니다. PyQt5/OpenCV 기반 실시간 화면, 클립 저장, 환자 정보 overlay, 보고서 생성 흐름도 README 기준으로 확인했습니다.

**R:** 로컬 README 기준으로 실시간 bounding box 렌더링, 스마트 클립 저장, AI 예비 보고서 생성, 환자 정보 관리 기능이 정리되어 있습니다. 포트폴리오 기존 문서에는 mAP 약 7%p 개선, 22 FPS 이상, 수상 이력이 포함되어 있습니다.

**R:** 의료 비전에서는 모델 구조의 새로움보다 데이터 특성과 평가 기준에 맞는 개선이 중요하다는 점을 배웠습니다.

**N:** 실제 임상 환경을 가정한 더 다양한 영상 조건, report 품질 평가, inference latency 로그를 추가하면 설득력이 더 높아집니다.

### 면접 1분 버전

RF-DETR 프로젝트에서는 대장 내시경 영상의 조명 변화와 장벽 왜곡 때문에 일반 객체 탐지만으로는 성능이 흔들리는 문제를 다뤘습니다. 저는 모델을 무작정 키우기보다 Grid Distortion과 Elastic Deform 같은 데이터 중심 증강으로 의료 영상의 왜곡을 반영하고, RF-DETR 기반 실시간 추론 파이프라인을 구성했습니다. README 기준으로 PyQt5/OpenCV 기반 실시간 렌더링, 스마트 클립 저장, 환자 정보 overlay까지 포함된 응용 형태로 정리되어 있습니다.

## 4. VAE 기반 비지도 유방암 병변 검출

### 포트폴리오 상세 버전

**S:** 유방 초음파 영상은 병변 라벨링 비용이 높아 충분한 지도학습 데이터를 확보하기 어렵습니다.

**T:** 정상 조직 데이터만으로 병변 후보를 찾아내는 비지도 이상 탐지 파이프라인을 구성해야 했습니다.

**A:** VAE로 정상 조직 분포를 학습하고, 입력 이미지와 재구성 이미지의 차이인 Reconstruction Error를 계산했습니다. 이미지별 노이즈 편차를 줄이기 위해 Dynamic Threshold 후처리를 적용했습니다.

**R:** 로컬 README 기준으로 max 90% Dice Coefficient와 2024년 공학혁신상 수상 내용이 확인됩니다. `vae_anomaly_detection.py` 단일 엔드투엔드 모듈도 로컬 저장소에 존재합니다.

**R:** 의료 AI에서는 모델 자체뿐 아니라 후처리와 평가 기준이 결과 품질을 크게 좌우한다는 점을 배웠습니다.

**N:** 데이터셋 분리, 재현 가능한 평가 스크립트, 사례별 FP/FN 분석을 추가하면 검증력이 높아집니다.

### 면접 1분 버전

VAE 프로젝트에서는 유방 초음파 병변 라벨이 부족한 문제를 정상 조직 기반 비지도 이상 탐지로 재정의했습니다. VAE가 정상 패턴을 복원하게 하고, 입력 이미지와 재구성 이미지의 차이를 Reconstruction Error Map으로 계산해 병변 후보를 찾았습니다. 특히 영상별 노이즈가 달라 고정 임계값이 불안정했기 때문에 Dynamic Threshold 후처리를 적용했습니다. 이 경험을 통해 의료 AI에서는 모델 구조만큼 후처리와 평가 기준이 중요하다는 점을 배웠습니다.

## 5. 블로그 자동화 파이프라인

### STAR-RN 요약

**S:** 부트캠프 학습 내용을 꾸준히 블로그로 정리해야 했지만, 매일 같은 품질과 형식을 유지하는 데 시간이 많이 들었습니다.

**T:** 간단한 메모를 기반으로 평가 기준과 필수 태그를 만족하는 기술 블로그 초안을 자동 생성해야 했습니다.

**A:** Gemini API 기반 생성 흐름, 제목/태그 규칙, 금지 내용 방지 프롬프트, SEO 키워드 반복 기준을 README에 정리했습니다.

**R:** `result/` 폴더에 날짜별 블로그 초안이 생성되는 구조를 마련했습니다. 다만 Git 저장소가 아니므로 구현 검증은 로컬 파일 기준입니다.

**R:** 자동화는 단순 생성보다 제약 조건과 품질 기준을 프롬프트에 명확히 넣는 것이 중요하다는 점을 배웠습니다.

**N:** 테스트 가능한 prompt regression set, 금칙어 검사, publishing workflow 자동화를 추가하면 완성도가 높아집니다.
