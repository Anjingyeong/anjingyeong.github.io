# 포트폴리오 제출용 점검 보고서

작성일: 2026-07-07

## 종합 판단

현재 포트폴리오는 AI 엔지니어 및 AI 기능 구현이 가능한 풀스택 개발자 지원용으로 사용할 수 있습니다. 다만 수정 전에는 일부 프로젝트 설명이 내부 분석 문장, 확장 후보, 검증 전 수치와 섞여 있어 "실제로 구현한 것"과 "앞으로 개선할 것"의 경계가 흐려질 위험이 있었습니다.

수정 후 방향은 다음과 같습니다.

- 첫 화면: 모델 자체보다 데이터 전처리, 평가 기준, 이벤트 메타데이터, 백엔드 연동까지 다루는 지원자로 정리.
- 프로젝트 카드: 기술 나열보다 문제와 구현 맥락이 먼저 보이도록 압축.
- 상세 모달: 문제 정의, 맡은 역할, 기술 선택 이유, 구현 내용, 성과, 한계 및 개선 방향이 드러나도록 문구 정리.
- 신뢰도: 완료되지 않은 대규모 확장 후보는 전면 노출에서 제외하고, 검증 수치가 불명확한 항목은 별도 검증 후 표기하도록 조정.

## P0: 반드시 수정해야 했던 문제

- RAG 프로젝트에 대규모 문서 관계 분석 확장 후보가 화면 노출 문구로 남아 있어 구현 완료처럼 오해될 수 있었습니다. 전면 표현에서 제거하고 현재 범위를 정적 JSON 인덱스와 하이브리드 검색 설계로 낮췄습니다.
- RF-DETR 프로젝트의 과거 설명에 22 FPS, mAP 등 화면 내 다른 문구와 충돌할 수 있는 수치가 남아 있었습니다. 웹 화면과 경험 섹션에서 검증 근거가 분명한 "실험 설정 기준 약 7% 개선" 중심으로 정리했습니다.
- Contact 폼이 EmailJS 미설정 시 브라우저 `alert()`를 띄웠습니다. 제출용 포트폴리오 UX에 맞게 버튼 상태 메시지로 안내하도록 수정했습니다.

## P1: 취업용 완성도를 높이는 개선

- Hero 문구를 "AI Engineer" 단독 강조에서 "모델을 서비스 흐름에 연결하고 운영자가 신뢰할 수 있는 AI 기능을 만드는 지원자"로 정리했습니다.
- 프로젝트 상세 모달 상단에 카드 요약 문구를 다시 보여줘, 핵심 요약에서 상세 근거로 이어지는 읽기 흐름을 만들었습니다.
- 프로젝트 카드를 키보드 `Enter`/`Space`로 열 수 있게 보강해 접근성과 사용성을 개선했습니다.
- README의 공개 소개와 기술 스택을 웹 포트폴리오의 현재 포지셔닝과 맞췄고, 내부 면접용 STAR-RN 문서 링크를 README 전면에서 제거했습니다.

## P2: 있으면 좋은 후속 개선

- 각 프로젝트에 실제 데모 링크 또는 짧은 시연 영상 링크를 추가하면 신뢰도가 더 올라갑니다.
- 스마트 안전 관제는 FPS, latency, 오탐/미탐 기준을 작은 표로 추가할 수 있을 때 가장 설득력이 커집니다.
- 의료 AI 프로젝트는 사용 데이터, split 기준, 평가 metric을 README 또는 GitHub 저장소에도 동일하게 반영하면 화면과 저장소 간 충돌 위험이 줄어듭니다.
- 모바일 모달에서 긴 다이어그램이 많은 프로젝트는 섹션 접기/펼치기를 도입하면 읽기 피로를 줄일 수 있습니다.

## 프로젝트별 제출용 2-3줄 요약

### 스마트 안전 관제 시스템

CCTV 영상에서 위험 상황 후보를 감지하고, AI 이벤트가 frameId, timestamp, cameraId, confidence 같은 증거 메타데이터와 함께 관제 화면까지 전달되도록 설계한 실시간 관제 보조 프로젝트입니다. RTSP, MQTT, WebSocket 흐름을 분리해 모델 결과가 알림과 운영 판단으로 이어지는 구조를 구현 관점에서 정리했습니다.

### RF-DETR 기반 대장 내시경 용종 검출 시스템

Kvasir 용종 데이터셋을 **Train 70% / Validation 20% / Test 10%**로 분할하고, RF-DETR 모델을 fine-tuning해 OpenCV GUI에서 카메라와 영상 파일 입력을 지원한 실시간 판독 보조 프로토타입입니다. 데이터 전처리, 데이터 증강, 모델 학습, 후보 모델/FPS 비교, bounding box 시각화를 담당했으며, 자동 녹화 기능은 담당 구현 범위로 표현하지 않습니다.

### VAE 기반 유방 초음파 이상 탐지

전문 라벨 확보가 어려운 유방 초음파 문제를 비지도 이상 탐지로 재정의하고, VAE 재구성 오차와 Dynamic Threshold로 병변 후보 영역을 찾은 프로젝트입니다. 데이터 전처리, Reconstruction Error Map 생성, 후처리 기반 오탐 감소 흐름을 담당했고 공학혁신상을 수상했습니다.

### LLM Wiki / RAG 기반 지식 관리

프로젝트 문서와 사고 기록을 정적 JSON 인덱스로 구조화하고, **BM25, Vector Search, Metadata Filtering, RRF**를 결합한 4단계 검색 흐름을 설계한 지식 관리 프로젝트입니다. 최종 코드뿐 아니라 실험 근거, 이벤트 메타데이터, 출처 정보를 다시 찾을 수 있게 만드는 데 초점을 뒀습니다.

- 지원동기: "저는 모델 성능만이 아니라 모델 결과가 실제 화면, 알림, 저장 구조, 운영 판단으로 이어지는 전체 흐름을 구현하는 AI 엔지니어로 성장하고 싶습니다."
- 경험: "의료영상 AI와 실시간 관제 프로젝트를 통해 데이터 전처리, 모델 fine-tuning, 평가 기준, 후처리, 이벤트 파이프라인을 함께 다뤘습니다."
- 강점: "도메인 문제를 모델 문제로만 보지 않고, 사용자가 판단할 수 있는 증거와 운영 흐름까지 연결해 설계하는 점이 강점입니다."
- 문제해결력: "오탐/미탐, threshold, 데이터 부족, 실시간 이벤트 동기화처럼 실제 서비스에서 발생하는 문제를 분리해 원인과 개선 방향을 정리했습니다."

## 남은 위험

- 스마트 안전 관제의 정량 지표는 아직 웹에 수치로 쓰지 않았습니다. 측정 로그가 확보되기 전까지는 완료 성과로 표기하지 않는 것이 안전합니다.
- GitHub 저장소 README와 포트폴리오 문구가 모두 같은 수준으로 최신화되었는지는 별도 저장소 확인이 필요합니다.
- Contact 폼은 EmailJS 환경변수가 설정된 배포 환경에서만 실제 메일 전송이 됩니다. 미설정 상태에서는 안내 상태가 표시됩니다.
