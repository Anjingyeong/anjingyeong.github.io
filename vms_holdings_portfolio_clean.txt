VMS 홀딩스 지원 첨부용: GitHub 프로젝트 기술 및 아키텍처 선택 배경

안녕하세요, VMS 홀딩스에 지원하는 풀스택 엔지니어입니다.
요청하신 바와 같이, 제가 진행한 주요 GitHub 프로젝트들의 '기술 선택의 이유'와 '아키텍처 설계 배경'을 VMS 홀딩스의 개발 방향성(빠른 MVP 검토, AI Agent 활용, 유연한 문제 해결)에 맞추어 정리했습니다.

저는 평소 Gemini, Cursor 등의 AI Coding Agent를 적극적으로 활용하여, 단순한 보일러플레이트 코드 작성이나 환경 세팅(ex. Dockerfile, docker-compose 구성 등)에 들어가는 리소스를 최소화하고 있습니다. 대신 그 시간을 '핵심 비즈니스 로직(서비스 문제) 해결'과 '트래픽/데이터 병목을 제어하는 아키텍처 설계'에 오롯이 투자하여 프로덕트의 품질과 속도를 동시에 높이는 데 집중합니다.


1. FullCount (실시간 스포츠 커뮤니티 및 에스크로 결제)
분야: 백엔드 아키텍처 및 REST API 설계
GitHub Link: https://github.com/Rookies5-MiniPj2-Team5

[왜 이 기술과 아키텍처를 선택했는가?] (VMS 핵심 역량: Spring Boot, REST API, RDB 스키마)

1) 기술 스택: Spring Boot, Spring Data JPA, MySQL
MVP의 빠른 구현과 데이터 무결성: 단기간 내에 핵심 서비스를 검증하는 MVP 모델 환경 하에서, 객체 지향적 도메인 설계를 빠르게 DB 스키마(RDB)에 매핑할 수 있는 Spring Data JPA를 채택했습니다. 특히 금전이 오가는 '에스크로 결제' 로직이 포함되어 있어, 데이터의 일관성(ACID)과 무결성을 보장하기 위해 NoSQL보다 MySQL 기반의 RDB 스키마 설계를 선택했습니다.

2) 아키텍처: WebSocket 통신 및 Redis Pub/Sub 도입
단방향 Polling의 한계 극복: 초기에는 프론트엔드가 백엔드 API를 지속적으로 호출(Polling)하도록 설계되었으나, 실시간 경기 중계 특성상 트래픽 스파이크 발생 시 DB에 막대한 I/O 오버헤드가 발생했습니다.
아키텍처 전환: 이를 해결하기 위해 단순 API 호출 구조를 버리고, 백엔드에 Redis와 WebSocket(STOMP)을 도입했습니다. 프론트엔드의 잦은 요청 부하를 줄이고, 상태 변경이 일어날 때만 클라이언트에 데이터를 밀어주는(Push) 이벤트 기반 아키텍처로 파이프라인을 재설계하여 전체 시스템의 지연과 병목 현상을 방지했습니다.


2. Lumina Capital (금융 데이터 파이프라인 및 서빙 대시보드)
분야: 빠르고 유연한 신규 MVP 개발 (R&D 성격)
GitHub Link: https://github.com/kimgeon0802/1team_mini_PJT

[왜 이 기술과 아키텍처를 선택했는가?] (VMS 핵심 역량: 시장 반응에 따른 유연한 MVP 설계)

1) 기술 스택: Python, Streamlit, MySQL
기술이 아닌 '서비스 문제'에 집중: 이 프로젝트의 본질은 방대한 금융 데이터를 수집하고 시각화하여 비즈니스 인사이트를 도출하는 것이었습니다. 복잡한 프론트엔드 프레임워크(React 등) 구조를 잡는데 리소스를 소모하기보다, 가장 빠르게 대시보드 형태의 MVP를 뽑아내고 검증하기 위해 Python과 Streamlit을 기술 스택으로 채택했습니다. 이는 기술 자체를 뽐내기보다 목적에 맞는 '빠른 실행력'을 우선한 결과입니다.

2) 아키텍처: 비동기(Asynchronous) 스케줄링 기반 캐시 파이프라인
타임아웃(Timeout) 붕괴 예방 요소: 외부 API로 데이터를 긁어와 웹에 바로 출력(동기식)하게 되면, 외부 서버의 지연이 그대로 우리 서비스의 렌더링 타임아웃으로 이어집니다.
Decoupling(분리) 설계: 프론트엔드 요청과 데이터 수집 사이의 의존성을 깨기 위해, 백그라운드 서버에서 CRON 스케줄러가 주기적으로 비동기 스크래핑을 수행해 DB에 적재(캐싱)만 해두고, 클라이언트는 언제든지 최적화된 DB만 즉시 읽도록(Read-only) 아키텍처를 분리했습니다. 외부 지연 요소로부터 프로덕트의 가용성을 완벽히 방어해냈습니다.


3. 의료 영상 실시간 병변 검출 AI 최적화 (학부 캡스톤)
분야: AI 모델 경량화 및 엣지 환경 배포 최적화
GitHub Link: https://github.com/anjin0910-afk/RF-DETR-project

[왜 이 기술과 아키텍처를 선택했는가?] (VMS 핵심 역량: 제약된 환경에서의 AI 배포/최적화)

1) 기술 스택: Python, PyTorch, RF-DETR
서비스 관점의 딥러닝 타겟팅: 단순 정확도 튜닝을 넘어, "이 AI가 실제 엣지 디바이스 환경에서 돌아갈 수 있는가?"라는 서비스 제약 조건 해결에 집중했습니다. 이를 위해 실시간 객체 탐지에 강점을 가지는 Transformer 기반의 RF-DETR 모델을 채택했습니다.

2) 아키텍처: 구조적 가지치기(Pruning)를 통한 추론 성능 최적화
인프라 제약을 뚫는 아키텍처: 최첨단 모델이라도 연산량이 가벼워야 상용화가 가능합니다. 제한된 하드웨어 리소스의 한계를 돌파하기 위해 무분별한 파라터를 덜어내는 '구조적 가지치기' 최적화를 적용했습니다. 결과적으로 모델을 덜어내면서도 초당 22 프레임의 실시간 추론 속도를 확보하며 대상(금상)을 수상했습니다. 향후 VMS에서 AI/LLM 기반 서비스 출시 시, 인프라 비용/속도 병목을 최적화할 수 있는 토대 경험입니다.


4. 유방암 검출 비지도학습 AI 파이프라인 설계
분야: AI/ML 저수준(Low-level) 파이프라인 제어
GitHub Link: https://github.com/anjin0910-afk/vae-breast-cancer-anomaly

[왜 이 기술과 아키텍처를 선택했는가?] (VMS 핵심 역량: LLM/AI 기반 서비스 파이프라인의 심층적 이해)

1) 기술 스택: Python, TensorFlow, VAE
블랙박스를 해체하는 엔지니어링: 프레임워크가 제공하는 손쉬운 함수(블랙박스) 호출 방식에만 의존해서는 미세한 병변(Anomaly)을 확실하게 군집화하기 어려웠습니다. 이에 텐서 연산의 저수준(Low-level) API를 직접 접근하고 다루기 위해 TensorFlow 파이프라인을 채택했습니다.

2) 아키텍처: 커스텀 손실 함수(Loss Function) 텐서 제어
표준 API의 한계 극복: 데이터 픽셀의 이질감을 유도하기 위해, KLD(분포 오차)와 MSE(재구성 오차)의 가중치를 데이터 특성에 맞게 직접 조작하는 커스텀 로스 함수 아키텍처를 설계했습니다. 
VMS에서 LLM(Gemini, GPT)을 서비스에 통합할 때에도, 단순 Prompt/API Wrapping 수준을 넘어 RAG 환경의 임베딩 성능 저하나 유사도 검색의 한계 같은, 근본적 AI 기술 이슈를 뜯어보고 효율적으로 최적화할 수 있는 귀중한 밑단 제어 역량입니다.


[Summary 결론]
위의 경험들을 통해, 저는 "Docker 등 환경 구성은 AI 에이전트의 힘을 빌려 빠르고 표준화되게 구성"하고, 사람(엔지니어)의 두뇌로는 "프론트-백엔드 간의 트래픽 병목을 풀고(Spring Boot/Redis), 데이터 지연을 끊어내며(Asynchronous), AI 서비스의 최적화 한계를 돌파하는 아키텍처 설계"에 집중하는 워크플로우를 완성했습니다. 잦은 피봇과 불확실성을 연료로 삼아 성장하는 VMS 홀딩스와 함께, 선제적인 아키텍처 설계로 비즈니스 속도를 견인하겠습니다.
